# RubiksCubeGPT

**Does a GPT trained *only* to predict solving moves build an internal model of the cube it never sees?**
A from-scratch replication of [**Othello-GPT**](https://arxiv.org/abs/2210.13382) on the Rubik's cube.

> A decoder-only Transformer is trained on nothing but move sequences: a scramble, then its
> solution. It is **never shown the cube's state or colors**. The question is whether a *world
> model* of the cube emerges spontaneously inside its activations. The evidence is twofold:
> **behavioral** (the model solves cubes it has never seen) and **internal** (a probe can read
> the cube's state back out of the frozen activations).
>

---

## Overview

The project follows the same idea as Othello-GPT, but applied to a much harder combinatorial object. A GPT trained only to predict the next move has no built-in reason to represent the cube : so if it turns out that it does, that representation had to be reconstructed purely from the sequence of moves.

The pipeline has four stages:

1. **Data generation** : scrambles + Kociemba solutions (deterministic), so the model learns (and we can check) where the cube actually stands at every step.
2. **Modeling** : a nanoGPT transformer model trained from scratch, with the loss masked so only the *solution* is learned
3. **Evaluation** : solve-rate reported **per scramble depth**
4. **Probing** : recovering the cube's internal state from the frozen residual stream


## Dataset

Generated by `scripts/generate_dataset.py`, which pairs a **homemade cube simulator**
(`rubikscubegpt/cube.py`, Kociemba's `cp/co/ep/eo` representation) with **Kociemba** as the
solving oracle.

Each example is a tuple of *(scramble, solution, tokens, per-token ground-truth state)*. Every
example is **verified**: the scramble is replayed and the Kociemba solution is checked to
actually solve the cube through the same simulator.

| Property | Value |
|---|---|
| Examples | ~100,000 |
| Scramble depth (moves) | 4 – 20 |
| Vocabulary | 21 tokens (18 moves + `<PAD>`, `<SEP>`, `<E>`) |
| Sequence length | ≤ 44 tokens |
| Train / validation split | ~90 / 10 |

> **Note on distribution.** Sampling the *scramble* depth uniformly concentrates the actual
> *difficulty* (i.e. the Kociemba **solution length** )  around 19–22 moves. Difficulty is a
> property of the state, not of how far you scrambled.

---

## Model

A standard **nanoGPT** decoder (`Head → MultiHeadAttention → FeedForward → Block →
GPTLanguageModel`), implemented from scratch in `rubikscubegpt/model.py`.

- **Architecture** : 8 layers, `n_embd = 256`, 4 heads, `block_size = 64`, dropout 0.2.
- **Masked loss** : the target is `-100` before `<SEP>` and on padding, so gradient flows only
  through the solution.
- **Schedule** : linear warmup then cosine decay to `min_lr`.
- **Save-best** : the checkpoint kept is the one at **minimum validation loss**, not the final
  (over-fitting) state.

## Results

### Behavioral - solve-rate

The model reaches a **~31 % global solve-rate**, but the more precise picture is **per depth**: a plateau around **50–65 % up to depth ~11**, followed by a
**cliff at 12–13**. `rubikscubegpt/eval.py` reports the solve-rate broken down by scramble depth.

### Internal - probing the world model

A **linear probe** on the frozen residual stream (`rubikscubegpt/probe.py`, sweeping every layer)
recovers the abstract cube state (`cp` / `co` / `ep` / `eo`) from the activations. The key
finding:

> **A world model emerges - but it is *structured by phase*, mirroring Kociemba's two-phase
> solver.**
>
> - **Orientation** (`eo` / `co`) is decoded **continuously**, from the scramble onward.
> - **Permutation** (`cp` / `ep`) is materialized **on demand**, only as the sequence approaches
>   the end-of-solution token `<E>`.

The interpretation is that the model represents *exactly what its solving policy needs, when it
needs it*  a structured, phase-aware world model rather than a full snapshot of the cube.


## Setup

Requires **Python 3.14**.

```bash
python3.14 -m venv .venv
.venv/bin/pip install torch kociemba numpy
```

Torch runs on MPS should be prefixed with `PYTORCH_ENABLE_MPS_FALLBACK=1`
(and `PYTHONUNBUFFERED=1` to watch the loss stream live).

---

## Running

```bash
# 1. Generate the dataset — train.txt + val.txt + dataset.json (with token-aligned ground truth)
.venv/bin/python -m scripts.generate_dataset

# 2. Train the model (MODE = "train") — or sample solutions (MODE = "generate")
PYTORCH_ENABLE_MPS_FALLBACK=1 .venv/bin/python -m rubikscubegpt.model

# 3. Evaluate — solve-rate per scramble depth on the validation set
PYTORCH_ENABLE_MPS_FALLBACK=1 .venv/bin/python -m rubikscubegpt.eval

# 4. Probe — read the cube state back out of the frozen activations, layer by layer
PYTORCH_ENABLE_MPS_FALLBACK=1 .venv/bin/python -m rubikscubegpt.probe
```

> Data generation runs with a verifier enabled: every scramble is replayed and its Kociemba
> solution is checked to actually solve the cube before the example is kept.
